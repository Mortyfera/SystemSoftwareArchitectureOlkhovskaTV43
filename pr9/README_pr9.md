sudo chown root:root pr93

**Пояснення**  
Програма демонструє механізм перемикання між реальним та ефективним користувачем. Після зниження привілеїв через seteuid() спроба змінити створений root-ом файл завершилася помилкою Permission denied, оскільки права 0644 дозволяють звичайному користувачу лише читання.

Створіть "обманну" програму, яка виглядає як ls, але виконує інші дії. Як її виявити?  
**Компіляція програми**  
gcc \-Wall pr913.c \-o pr913  
**Запуск програми**  
./pr913  
**Результат роботи програми**  
Custom ls  
pr95.c  pr94.c  ..  pr913.c  pr92.c  pr93.c  pr93  pr92  pr913  pr96  pr96.c  pr91  .  pr95  pr91.c  pr94  

\[INFO\] This is not real ls.  
**Пояснення**  
Для виявлення “обманних” програм слід використовувати чотири основні методи:

* Перевірка шляху: Команди which або type \-a покажуть, звідки саме запускається файл. Якщо це не /bin чи /usr/bin, перед вами підробка.  
* Аналіз аліасів: Команда alias викриє спроби підмінити назву утиліти на сторонній скрипт.  
* Абсолютний шлях: Запуск через /bin/ls гарантує виконання оригіналу, ігноруючи маніпуляції зі змінною PATH.  
* Хешування: Перевірка цілісності через sha256sum або утиліту debsums виявить будь-які зміни у бінарному файлі.

Розроблена утиліта наочно демонструє таку вразливість. Використовуючи dirent.h, вона імітує роботу ls, але додає кастомний текст, що дозволяє легко ідентифікувати підміну. Цей експеримент підтверджує, що маніпуляції з оточенням PATH або створення локальних копій програм можуть скомпрометувати безпеку системи, і підкреслює критичну важливість методів верифікації цілісності системних утиліт.
