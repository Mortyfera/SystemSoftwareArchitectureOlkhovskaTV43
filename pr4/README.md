#### **Завдання 4.1**

Скільки пам’яті може виділити malloc(3) за один виклик?

Параметр malloc(3) є цілим числом типу даних size\_t, тому логічно максимальне число, яке можна передати як параметр malloc(3), — це максимальне значення size\_t на платформі (sizeof(size\_t)). У 64-бітній Linux size\_t становить 8 байтів, тобто 8 \* 8 \= 64 біти. Відповідно, максимальний обсяг пам’яті, який може бути виділений за один виклик malloc(3), дорівнює 2^64. Спробуйте запустити код на x86\_64 та x86. Чому теоретично максимальний обсяг складає 8 ексабайт, а не 16?

**Рішення**

Теоретичний розмір пам'яті, який можна адресувати у 64-бітній системі, становить 2^64 байтів (16 ексабайт). Проте максимальний обсяг, який може виділити malloc(3) за один виклик, обмежений 8 ексабайтами (2^63 байтів). Зменшення ліміту вдвічі зумовлене двома основними факторами:

1. Обмеження стандарту мови C: Для обчислення відстані між вказівниками використовується знаковий тип даних ptrdiff\_t. Оскільки один біт із 64 виділяється під знак (для руху масивом вперед і назад), максимальне позитивне значення дорівнює 2^63-1 (8 ЕБ). Стандартні бібліотеки обмежують розмір блоку значенням PTRDIFF\_MAX, щоб уникнути переповнення під час навігації по пам'яті.  
2. Архітектура операційної системи: Віртуальний адресний простір системи традиційно ділиться навпіл. Одна половина (User Space) виділяється під програми користувача (тут працює malloc), а інша \- жорстко зарезервована для ядра ОС та драйверів (Kernel Space).

Порівняння архітектур:

* x86\_64 (64-біт): Загальний простір \- 16 ЕБ. Ліміт malloc \- 8 ЕБ.  
* x86 (32-біт): Загальний простір \- 4 ГБ. Ліміт malloc \- 2 ГБ.

#### **Завдання 4.4**

Чи є помилки у такому коді?

void \*ptr \= NULL;  
while (\<some-condition-is-true\>) {  
    if (\!ptr)  
        ptr \= malloc(n);  
    \[... \<використання 'ptr'\> ...\]  
    free(ptr);  
}

Напишіть тестовий випадок, який продемонструє проблему та правильний варіант коду.

**Рішення**

У наданому фрагменті кодує наступні помилки: утворення “завислого вказівника” (dangling pointer), використання пам'яті після звільнення та подвійне звільнення.

Причина “Aborted (core dumped)”: На першій ітерації пам'ять коректно виділяється та звільняється функцією free(ptr). Проте сама змінна ptr не очищається і продовжує зберігати стару адресу. Через це на другій ітерації умова if (\!ptr) виявляється хибною, нове виділення пам'яті (malloc) пропускається, і програма намагається повторно використати та звільнити ту саму адресу. Це призводить до аварійного переривання програми операційною системою.

Виправлення: Для забезпечення коректної роботи алгоритму необхідно примусово скидати вказівник одразу після звільнення пам'яті, додавши рядок ptr \= NULL;. Це гарантує, що на наступній ітерації цикл коректно виділить новий блок пам'яті.
