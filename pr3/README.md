У Linux існують м'які та жорсткі ліміти ресурсів. Головна особливість полягає в тому, що звичайний користувач може змінювати ліміти лише у бік зменшення, тому, випадково опустивши жорсткий ліміт, він потрапляє у незворотну ситуацію. Для Docker це критично важливо: якщо процес всередині контейнера помилково затисне собі ліміт відкритих файлів, високонавантажені додатки (як-от бази даних) почнуть падати з помилкою “Too many open files”, і виправити це можна буде лише повним перезапуском контейнера з новими налаштуваннями.
```
root@508e9b264cc4:/\# ulimit \-n  
1024  
root@508e9b264cc4:/\# ulimit \-Sn  
1024  
root@508e9b264cc4:/\# ulimit \-Hn  
524288  
root@508e9b264cc4:/\# ulimit \-n 3000  
root@508e9b264cc4:/\# ulimit \-Sn  
3000  
root@508e9b264cc4:/\# ulimit \-Hn  
3000  
root@508e9b264cc4:/\# ulimit \-n 3001  
bash: ulimit: open files: cannot modify limit: Operation not permitted  
root@508e9b264cc4:/\# ulimit \-n 2000  
root@508e9b264cc4:/\# ulimit \-n  
2000  
root@508e9b264cc4:/\# ulimit \-Sn  
2000  
root@508e9b264cc4:/\# ulimit \-Hn  
2000  
root@508e9b264cc4:/\# ulimit \-n 3000  
bash: ulimit: open files: cannot modify limit: Operation not permitted
```
Користувач root усередині Docker-контейнера є суттєво обмеженим суперкористувачем. З міркувань безпеки та для захисту хост-системи від збоїв, Docker за замовчуванням позбавляє його низки привілеїв ядра Linux, зокрема дозволу CAP\_SYS\_RESOURCE. Через відсутність цього права контейнерний root не може підвищувати жорсткі ліміти ресурсів, тому, випадково знизивши їхню “стелю”, він опиняється в тій самій незворотній ситуації, що й звичайний користувач.
